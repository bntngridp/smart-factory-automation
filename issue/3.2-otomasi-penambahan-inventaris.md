# 3.2 Otomasi Penambahan Inventaris (Trigger Logic)

**Epic:** 3 — Tracking Produksi & Pergerakan Inventaris  
**Priority:** High  
**Dependencies:** 3.1, 1.1

---

## Deskripsi

Mengimplementasikan logika backend di mana setiap kali data masuk ke `ProductionLogs`, sistem otomatis membuat record baru di `InventoryMovements` dengan tipe "IN".

## Acceptance Criteria

- [ ] Saat `createProductionLog` dipanggil, sistem secara otomatis:
  1. Menyimpan data ke tabel `ProductionLogs`.
  2. Membuat record baru di `InventoryMovements` dengan:
     - `productId` — ID produk yang diproduksi.
     - `quantity` — jumlah yang sama dengan log produksi.
     - `type` — `"IN"`.
     - `reference` — ID dari production log (untuk tracing).
  3. Kedua operasi dibungkus dalam **Prisma Transaction** — jika salah satu gagal, kedua operasi dibatalkan (rollback).
- [ ] Tidak ada data ganda atau inkonsistensi antara ProductionLogs dan InventoryMovements.
- [ ] Fungsi `createProductionLog` di-export dari `src/services/productionService.ts`.

## Technical Notes

- Pattern Prisma Transaction:

```ts
export async function createProductionLog(data: ProductionLogInput) {
  return prisma.$transaction(async (tx) => {
    const log = await tx.productionLog.create({
      data: {
        productId: data.productId,
        quantity: data.quantity,
        operatorName: data.operatorName,
      },
    })

    await tx.inventoryMovement.create({
      data: {
        productId: data.productId,
        quantity: data.quantity,
        type: "IN",
        reference: log.id,
      },
    })

    return log
  })
}
```

- Tidak menggunakan database trigger SQL — logika ada di aplikasi (Next.js Server Action).
- `revalidatePath` setelah transaksi sukses.

## Definition of Done

- Input 500 unit "Baut Industri M8" di form produksi → `InventoryMovements` bertambah 500 unit untuk produk yang sama.
- Transaksi atomic: jika insert InventoryMovement gagal, ProductionLog juga tidak tersimpan.
- Data konsisten dan dapat diverifikasi melalui database.
